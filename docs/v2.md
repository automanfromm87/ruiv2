# rui v2 设计：代码即基础设施的同构云平台框架

> 状态：设计草案（v2 target architecture）。本文承接 `docs/arch.md` 的 5 层愿景，向"代码即基础设施"演进。

> **诚实前置（必读）**：v2 的**运行时地基是已落地的真实代码**——L1–L3 同构内核、`DbExecutor`/`set_db_executor` 依赖倒置（`crates/rui/src/gql/orm.rs`）、type-level `Field` 投影（`gql/mod.rs`）、`with_request_runtime` 隔离（`server.rs`）、`router!`/`#[page]`/`#[derive(Ent)]`（`rui-macros`）、`App`/`AppConfig`（~4500 LOC，commit `de55deb..b82215d`）。但 v2 **新增的一切**——三类 Executor、`Ctx`/统一端点、`platform!{}`、`rui build/describe/deploy`、`CloudProvider`——**在当前仓库里尚未实现**，本文是**设计**而非现状描述。凡涉及未实现部分均用"将/需"而非"是"。`docs/arch.md` 里的 Platform/多 Host 段落（web/job/cron/grpc 共享 App Core）是**愿景草图**，不是已实现代码。
>
> **承诺范围切分**（详见 §9.2）：本文区分 **v2 deliverable**（第一版承诺、有清晰落地路径：三 Executor + Ent 扩 cache/queue + 统一端点 + `platform!{}` + `rui describe` 出 DAG/JSON + `rui deploy` 单云推镜像连外置资源）与 **v2 vision**（方向已定但不在第一版承诺：跨 crate type-level 依赖边自动推断、多云对称、自研 provision 有状态资源、DB migration 自动化）。凡标 vision 处只给方向与接口骨架，并诚实标注未解难点。

## 0. TL;DR

### 0.1 一句话定位

**rui v2 = 一个同构 Rust 全栈框架 + 一个"代码即基础设施"的编译期部署系统。** 你用统一的原语（`Ent` / `Service` / `Route` / `Job` / `Cron`）写应用，框架在编译期把整个应用的拓扑（计算单元 + 存储资源 + 它们之间的依赖边）提取成一张 **deployment DAG**，然后一键 provision 并部署到抽象的 `CloudProvider`（AWS / GCP）。

### 0.2 与 v1 的关系

rui v1 已经证明了"下半身"：细粒度响应式内核、编译期校验的 GraphQL 数据层、Relay 规范化缓存、SSR + 水合、GraphQL-native ORM（`#[derive(Ent)]` 单一真相源 + selection 投影下推 + `DbExecutor` 依赖倒置）、可换入换出的 `RequestRuntime`、4 层分层 gate、`AppConfig` 可配置宿主。

v1 的局限在"上半身"和"身体之外"：

| 维度 | v1 现状 | v2 目标 |
|---|---|---|
| 存储 | 只有 DB（单实体、Eq/Contains、内存分页） | `Ent` 统一抽象 DB + Cache + MessageQueue，带 edges/relations |
| 计算 | 只有 HTTP host（SSR/CSR/API 混在 router! + /graphql） | 统一的 `Route`/`Service`/`Job`/`Cron` 四原语，一致接口 |
| 部署 | 手写 `bin/ssr.rs` + 手设环境变量 + 无部署故事 | 编译期提取 DAG → 一键部署到 CloudProvider |
| 业务复用 | resolver / handler 各写一遍 | App Core use case 只写一次，所有 host 薄壳复用 |

**关键连续性**：v2 不是重写，而是把 v1 已经验证的 4 个 seam —— `set_db_executor`/`set_transport` 依赖倒置、`with_request_runtime` 隔离、`app!`/`router!` 显式装配 registry、`AppConfig` 宿主配置 —— **从"内部实现细节"提升为"架构主梁"**，并在其上长出 Cache/Queue 存储、统一计算原语、和编译期部署系统。四条不变量（零依赖默认 / 4 层分层 gate / 同构 / RequestRuntime 隔离）全程不破。

## 1. 动机与目标

### 1.1 消除"转化税"

v1 的核心痛点（memory: graphql-native-orm-direction）是**转化税**：同一个概念要在多个层手写转换。v1 已经用 `#[derive(Ent)]` 消掉了"三重 derive"（GqlObject + SimpleObject + FromRow）和"双 schema"。但更大的转化税还在：

1. **存储转化税**：Cache 和 Queue 今天根本不存在，开发者要自己 `lazy_static` 一个 `HashMap` 当缓存、自己写线程 + `mpsc` 当队列，且这些都游离在框架的类型系统和 GraphQL 之外。
2. **计算转化税**：一个"用户注册"业务，要在 GraphQL resolver 里写一遍、在后台 job 里写一遍、在 cron 里再写一遍。
3. **部署转化税**：写完代码后，要手写 Dockerfile、手写 Terraform、手在 AWS 控制台连 RDS/SQS/ElastiCache、手把连接串塞进环境变量 —— 而这些信息**本来就 100% 蕴含在代码里**。

### 1.2 代码即基础设施（Infrastructure from Code）

rui v2 的赌注：**因为所有关键组件都是我们自己的原语，编译器可以完整地知道应用的基础设施拓扑。** 你声明 `Cache<User>`，编译器就知道"要 provision 一个缓存"；你在 `Service` 里持有一个 `Queue<EmailJob>` 字段，编译器就知道"这个 service 依赖那个队列"。这条边不需要你写 YAML，它就是你的 Rust 类型。

这区别于 Terraform（基础设施与代码分离、手工对齐）和 K8s YAML（与代码分离）。最接近的先例是 **Encore.dev**，但 rui 用 Rust 的类型系统 + 显式装配宏来做静态提取，而非独立的源码解析器。

### 1.3 统一心智模型

开发者只需要学**一套**定义方式。`Route` / `Service` / `Job` / `Cron` 都是"带类型化输入、依赖注入的 `ctx`、返回类型化输出的处理单元"，区别只在**触发器**（HTTP 请求 / 进程内调用 / 队列消息 / cron 时刻）。`Ent` / `Cache` / `Queue` 都是"schema-as-code 的资源声明"。学会一个，就学会了全部。

### 1.4 目标与非目标

**目标**：(1) Ent 存储统一抽象 + 默认 GraphQL；(2) 四原语统一接口；(3) 编译期 DAG 提取 + 一键部署。

**非目标（v2 第一版）**：不做多云同时部署、不做 service mesh、不做 async 引擎重写（见 §9 范围收敛）、不做可视化拓扑编辑器。

## 2. v2 总体架构

### 2.1 分层图

v2 在 v1 的 5 层之上，**新增 3 个关注点**（App Core、Compute Hosts、Infra Resources），并把"部署"作为一个**编译期 + 命令行**的横切系统。注意：新增的全部是 native-only 的上层，**Kernel/UI/Data 三层的同构性和零依赖性完全不动**。

```
  ┌──────────────────────────────────────────────────────────────────┐
  │  APPLICATION (用户代码)                                              │
  │   Ent/Cache/Queue 声明 · Service · Route(SSR/CSR/API) · Job · Cron  │
  │   App Core use cases(业务只写一次) · platform!{} 装配               │
  └───────────────┬──────────────────────────────────┬───────────────┘
                  │ 被宏读取                           │ 运行时调用
                  ▼                                    ▼
  ┌───────────────────────────────┐   ┌──────────────────────────────┐
  │ COMPILE-TIME SYSTEM (rui-macros)│  │  COMPUTE HOSTS (rui-platform) │
  │  view!/page/component/query!… │   │  WebHost / WorkerHost /       │
  │  #[ent]/#[service]/#[route]/  │   │  CronHost / ServiceHost       │
  │  #[job]/#[cron]               │   │  (都复用 serve/serve_axum 装配)│
  │  platform!{} → AppModel IR    │   └───────────────┬──────────────┘
  └───────────────┬───────────────┘                   │
                  │ 产出 IR                            │ 依赖
                  ▼                                    ▼
  ┌───────────────────────────────┐   ┌──────────────────────────────┐
  │  rui build → AppModel(DAG)    │   │  INFRA RESOURCES (rui-ent)    │
  │  rui deploy → ProvisionPlan   │   │  StoreExecutor/CacheExecutor/ │
  │       (rui-cli + rui-deploy)  │   │  QueueExecutor (依赖倒置)      │
  └───────────────┬───────────────┘   └───────────────┬──────────────┘
                  │ 映射                                │ 注入(set_*_executor)
                  ▼                                    ▼
  ┌──────────────────────────────────────────────────────────────────┐
  │  CLOUDPROVIDER (rui-cloud + rui-cloud-aws / rui-cloud-gcp)         │
  │   provision/diff/apply/destroy · 资源映射 · secrets · 运行时绑定    │
  └──────────────────────────────────────────────────────────────────┘

  ── 以下为 v1 已有、v2 不动的同构内核(零依赖默认) ──
  ┌──────────────────────────────────────────────────────────────────┐
  │ DATA RUNTIME (gql: exec/store/orm/value)  ·  UI RUNTIME (view/…)   │
  │ REACTIVE KERNEL (signal/effect/memo/scope/context)                 │
  │ HOST ADAPTERS (browser dom · ssr · std server · axum)              │
  └──────────────────────────────────────────────────────────────────┘
```

### 2.2 依赖方向（守住 4 层 gate）

依赖**只能向下 / 向抽象**，由 `crates/rui/tests/layering.rs` 强制：

- Kernel（reactive/props）零 `crate::` 依赖 —— 不动。
- Data Runtime（gql）不 NAME host、不依赖上层 —— Cache/Queue 的**接口** trait 放这层（与 `DbExecutor`/`Transport` 同级），只持纯接口。
- Compute Hosts / Infra 具体实现（PgExecutor、RedisCache、SqsQueue）是 native-only、optional、由 host 在启动时注入 —— 与 v1 的 `PgExecutor` 同构。
- CloudProvider（rui-cloud-*）是独立 optional crate，**编译 rui 本体时根本不参与**，只被 `rui-cli` 在部署时链接。

**零依赖默认铁律**：`rui` 的依赖树仍然只有 `rui-macros`。axum/tokio/redis/aws-sdk 全部 `optional` + `target.'cfg(not(wasm32))'`。wasm 客户端一个都不拉。

### 2.3 五大关注点如何各就各位

| 关注点 | 落点 | 复用的 v1 seam |
|---|---|---|
| **App Core**（业务 use case） | 普通 Rust struct/impl，不依赖任何 host | —— |
| **多 Host**（Web/Worker/Cron/Service） | `rui-platform`，每个 host 是薄壳 | `serve`/`serve_axum` 装配点、`with_request_runtime` |
| **Infra 资源**（DB/Cache/Queue） | `rui-ent` 接口 + native executor | `set_db_executor` 依赖倒置模式推广 |
| **Compile Model** | `rui-macros` 的 `platform!{}` → IR | `app!`/`router!` 显式装配哲学 |
| **CloudProvider** | `rui-cloud-*` + `rui-cli` | `AppConfig` 配置注入模式 |

### 2.4 守住的四条不变量

1. **零依赖默认**：core 仍只依赖 `rui-macros`；所有云/存储/async 依赖 optional + target-gated。
2. **4 层分层 gate**：新接口（CacheExecutor/QueueExecutor）进 Data 层、不 NAME 驱动；新实现进 host 层。`layering.rs` 加断言覆盖。
3. **同构**：`Ent`/`Cache`/`Queue` 的**类型层**（schema、payload 类型）wasm+native 共享；executor 实现 native-only（`#[cfg(not(wasm32))]`）。`Route` 的渲染同构；`Service`/`Job`/`Cron` 是 native-only 计算。
4. **RequestRuntime 隔离**：Web host 每请求 `with_request_runtime`；Worker/Cron host 每个 job 执行也包一层等价的 `with_job_runtime`，保证 thread-local 状态隔离、panic 安全。

## 3. Pillar 1 — Ent 存储框架（DB / Cache / MessageQueue）

### 3.1 设计目标与现状基线

v1 的 `#[derive(Ent)]`（`crates/rui/src/gql/orm.rs` + `rui-macros`）已经确立了**单一真相源**的范式：一个 struct 同时是 GraphQL 对象（同构）+ 表映射（native `SqlEntity`），selection 驱动列投影，`DbExecutor` 经 `set_db_executor` 依赖倒置注入。v2 把这个范式**从"DB 单一存储"推广到"DB + Cache + MessageQueue 三类存储"**，并补齐 entgo.io 级别的 schema 能力（edges / relations / mixins / hooks）。

**核心洞察**：v1 的 `DbExecutor` 依赖倒置 trait（L3 只持纯接口、host 注入具体驱动）是一个可以**机械复制三次**的模式。Cache 和 Queue 各自就是"另一个 `DbExecutor`"。

### 3.2 三类 Executor：依赖倒置 trait（Data 层，native-only）

沿用 `orm.rs` 中 `DbExecutor` 的形态（`fetch(&Query)` / `write(&Write)` + `OnceLock` 注入），新增两个对等 trait。三者都遵守"L3 不 NAME 任何驱动"的分层 gate：

```rust
// crates/rui/src/gql/orm.rs —— 已存在，v2 扩展 Query/Write 支持 join
pub trait StoreExecutor: Send + Sync {
    fn fetch(&self, q: &Query) -> Vec<Vec<(String, Value)>>;
    fn write(&self, w: &Write) -> WriteResult;        // v2: 返回受影响行 / 自增 id
    fn fetch_batch(&self, qs: &[Query]) -> Vec<Vec<Vec<(String, Value)>>>; // v2: DataLoader 批量
}

// crates/rui/src/store_kv.rs(新增，Data 层接口)
pub trait CacheExecutor: Send + Sync {
    fn get(&self, ns: &str, key: &str) -> Option<Vec<u8>>;
    fn set(&self, ns: &str, key: &str, val: &[u8], ttl: Option<Duration>);
    fn del(&self, ns: &str, key: &str);
}

// crates/rui/src/queue.rs(新增，Data 层接口)
pub trait QueueExecutor: Send + Sync {
    /// 发布一条消息到 topic;返回 message id。payload 已由上层序列化为字节。
    fn publish(&self, topic: &str, payload: &[u8], opts: &PublishOpts) -> String;
    /// 拉取(pull 模型;push 模型由 host adapter 直接回调 dispatch,不走这里)。
    fn pull(&self, topic: &str, max: usize) -> Vec<QueueMsg>;
    fn ack(&self, topic: &str, receipt: &str);
}

// 注入点 —— 与 set_db_executor 完全同构
pub fn set_store_executor(e: Box<dyn StoreExecutor>);
pub fn set_cache_executor(e: Box<dyn CacheExecutor>);
pub fn set_queue_executor(e: Box<dyn QueueExecutor>);
```

> **分层 gate**：这三个 trait 只出现在 Data 层，签名里**不出现** redis/sqs/postgres 任何字样。`RedisCache`/`SqsQueue`/`PgStore` 全在 host 层（optional crate），启动时 `set_*_executor` 注入。`layering.rs` 增加断言：`store_kv.rs`/`queue.rs` 不得 `NAME` 任何驱动。

### 3.3 Ent schema-as-code：`#[ent]` 宏

v1 的 `#[derive(Ent)]` 是"字段即列"。v2 升级为 entgo 风格的**显式 schema**，但保持 Rust 惯用法（不引入 entgo 的 codegen 二段式）。关键：schema 声明本身是**同构的纯类型**（wasm+native 都能编译），只有 executor 是 native。

```rust
// app/src/domain/user.rs
#[ent(table = "users", cache = "user_cache")]   // cache=.. 声明实体级缓存(可选)
pub struct User {
    #[ent(pk)]
    pub id: Id<User>,
    #[ent(index, unique)]
    pub email: String,
    pub name: String,
    #[ent(default = "now")]
    pub created_at: Timestamp,

    // ── edges/relations ──
    #[edge(has_many = Order, via = "user_id")]
    pub orders: Edge<Order>,           // O2M
    #[edge(many_to_many = Role, through = "user_roles")]
    pub roles: Edge<Role>,             // M2M
}

// mixins:用普通 trait + 派生组合,而非魔法
#[ent(table = "orders", mixins(SoftDelete, Timestamps))]
pub struct Order { #[ent(pk)] pub id: Id<Order>, pub user_id: Id<User>, pub amount_cents: i64 }
```

`#[ent]` 在 **native** 端生成（向后兼容 v1 的 `SqlEntity`，是其超集）：

```rust
impl SqlEntity for User {
    const TABLE: &'static str = "users";
    const PK: &'static str = "id";
    const COLUMNS: &'static [&'static str] = &["id","email","name","created_at"];
    const EDGES: &'static [EdgeDef] = &[                  // v2 新增
        EdgeDef { name:"orders", kind:HasMany, target:"orders", via:"user_id" },
        EdgeDef { name:"roles",  kind:M2M,     target:"roles",  through:"user_roles" },
    ];
    const CACHE: Option<&'static str> = Some("user_cache");  // → DAG 边:User → user_cache
}
```

在 **同构** 端生成 GraphQL 对象 + Relay `UserConnection/UserEdge/UserPageInfo`（v1 已有，复用 `emit_gql_object`/`emit_relay_value_object`）。**edges 自动成为 GraphQL 嵌套字段**：`user { orders { id amount_cents } }` 合法。

### 3.4 关系 + N+1：DataLoader（编译期已知边 → 自动批量）

因为 `EDGES` 是编译期常量，resolver 解析嵌套字段时，框架自动走 **per-request DataLoader**（挂在 `RequestRuntime` 上，天然请求级隔离）：

```rust
// 伪代码:exec.rs 解析 user.orders 时
// 1) 收集本层所有 User 的 id     → [u1, u2, u3]
// 2) 一次 fetch_batch:Query{ table:"orders", filter: In("user_id", [u1,u2,u3]) }
// 3) 按 user_id 分组回填           → 消除 N+1
```

`Filter` 枚举（`orm.rs`）从 v1 的 `Eq/Contains` 扩展：新增 `In(col, Vec<Scalar>)`（DataLoader 必需）、`Gt/Lt`（keyset 分页）。`Query` 新增 `joins: Vec<JoinSpec>`（M2M through 表）。**投影下推不变**：edges 字段不在实体列里，触发 DataLoader 而非列投影；标量字段仍走 `columns_for`。

### 3.5 hooks：写前后拦截

hooks 是普通的注册函数，挂在实体类型上，由 `write()` 路径调用（native-only）：

```rust
#[ent_hooks(User)]
impl UserHooks {
    fn before_create(u: &mut User, ctx: &EntCtx) -> Result<(), EntError> {
        u.email = u.email.to_lowercase(); Ok(())
    }
    fn after_create(u: &User, ctx: &EntCtx) {
        ctx.cache().del("user_cache", &u.id.to_string());   // 失效缓存
        ctx.queue::<UserCreated>().publish(UserCreated { id: u.id }); // 发事件
    }
}
```

> 注意 `after_create` 里 `ctx.queue::<UserCreated>()` —— 这就是一条**编译期可见的依赖边**（User 写路径 → UserCreated topic），见 §5。

### 3.6 Cache 资源

Cache 是一个**声明式资源**，不是裸 trait 调用。声明即 DAG 节点：

```rust
// 声明一个缓存资源(keyspace + 值类型 + 默认 TTL)
#[cache(name = "user_cache", ttl = "5m")]
pub struct UserCache(Cache<Id<User>, User>);   // 类型化 K/V

// 用法:typed handle,内部走 CacheExecutor(序列化用 gql::value,零额外依赖)
let cache: Cache<Id<User>, User> = ctx.cache::<UserCache>();
let u = cache.get_or_load(id, || store.fetch_one::<User>(id)).await?; // 读穿透
```

**与 Ent 的关系**：两种缓存策略并存 —— (a) 实体级缓存（`#[ent(cache=..)]`，按 PK 自动读穿透 + hook 失效）；(b) 自由 KV 缓存（`#[cache]` 资源，手动 get/set，用于会话、限流计数等）。

### 3.7 MessageQueue 资源

Queue/Topic 也是声明式资源，**消息类型即契约**：

```rust
#[queue(name = "emails", ordering = false, dedupe = true)]
pub struct EmailTopic(Topic<SendEmail>);  // Topic<P> 的 P 是同构可序列化 payload

#[derive(Serialize, Deserialize)]   // 注:rui 自带 gql::value 编解码,亦可 serde(optional)
pub struct SendEmail { pub user_id: Id<User>, pub kind: EmailKind }

// 生产者:typed publish
ctx.queue::<EmailTopic>().publish(SendEmail { user_id, kind: Welcome }).await?;
```

Topic 的**消费者就是 Pillar 2 的 `AsyncJob`**：一个 `#[job]` 声明它消费哪个 topic，框架据此在 DAG 里连边（topic → job worker），并在部署时为该 topic 接上订阅（SQS consumer / Pub-Sub push）。详见 §4。

### 3.8 GraphQL 默认暴露（不变）

Ent → GraphQL 的暴露完全复用 v1：`#[ent]` 同构生成对象类型 + Connection。v2 只新增"edges 字段进 selection、投影下推对标量、DataLoader 对关系"。Cache/Queue **默认不进 GraphQL**（它们是基础设施，不是查询面），但 queue 的 publish 可被 mutation resolver 调用。

### 3.9 每个资源即 DAG 节点（接 §5 的钩子）

关键的部署关联：`#[ent]`/`#[cache]`/`#[queue]` 每个宏都在 native 端 emit 一条 **`inventory::submit!` 资源描述**（或等价的 `linkme` slice 项）：

```rust
// #[ent(table="users", cache="user_cache")] 额外 emit:
rui::inventory::submit! { ResourceNode::Ent { name:"User", table:"users", cache:Some("user_cache") } }
// #[cache(name="user_cache", ttl="5m")] emit:
rui::inventory::submit! { ResourceNode::Cache { name:"user_cache", ttl_secs:300 } }
// #[queue(name="emails", ...)] emit:
rui::inventory::submit! { ResourceNode::Queue { name:"emails", ordering:false, dedupe:true } }
```

这些是 §5 编译期 DAG 提取的"叶子事实"。**向后兼容**：v1 `#[derive(Ent)]` 保留为 `#[ent]` 的最小子集别名（只生成 `SqlEntity` + GraphQL，不 emit edges/resource node），现有 examples/stocks 不改也能编。

## 4. Pillar 2 — 统一 API / 计算层

### 4.1 核心 KPI：四原语，一套心智模型

用户的硬性要求：`Route`（SSR/CSR/API）、`AsyncJob`、`CronJob`、`Service` 的定义接口**统一一致**。v2 的回答是：它们都是同一个抽象 `Handler` 的特化 —— **一个带类型化输入 `In`、依赖注入上下文 `Ctx`、返回类型化输出 `Out` 的处理单元**，区别只在**触发器（Trigger）**。

```rust
// crates/rui-platform/src/handler.rs —— 统一内核
pub trait Handler {
    type In;                    // 输入:HTTP body / job payload / cron tick / rpc req
    type Out;                   // 输出:Html / Json / () / rpc resp
    type Trigger: TriggerKind;  // 触发器种类(编译期标记)
    fn call(ctx: &Ctx, input: Self::In) -> Result<Self::Out, AppError>;
}
```

四个原语都是这个 trait 的"风味"，由**同名风格的属性宏**生成 `Handler` impl + 一条 inventory 节点描述。**接口一致性体现在**：同样的 `ctx`、同样的 `Result<_, AppError>`、同样的依赖注入、同样的 `In -> Out` 形状。

### 4.2 并排看一致性

```rust
// ── Route(API)：触发器 = HTTP，Out = Json ──
#[route(api, "POST /orders")]
async fn create_order(ctx: &Ctx, input: CreateOrderInput) -> Result<Json<OrderId>, AppError> {
    Ok(Json(ctx.app::<OrderService>().create(input).await?))
}

// ── Route(SSR)：触发器 = HTTP，Out = Html(复用 v1 page/Strategy) ──
#[route(ssr, "/orders/:id")]
async fn order_page(ctx: &Ctx, p: PathParams) -> Result<Html, AppError> {
    let o = ctx.app::<OrderService>().get(p.id).await?;
    Ok(Html(view! { <OrderView order=o/> }))
}

// ── Service：触发器 = 进程内/RPC，In/Out = 方法签名 ──
#[service]
impl OrderService {
    fn deps(c: &Ctx) -> Self { Self { db: c.store::<Order>(), jobs: c.queue::<EmailTopic>() } }
    async fn create(&self, input: CreateOrderInput) -> Result<OrderId, AppError> { /* 业务 */ }
}

// ── AsyncJob：触发器 = 队列消息，In = payload ──
#[job(topic = EmailTopic, retries = 5, dedupe_key = "welcome:{user_id}")]
async fn send_email(ctx: &Ctx, msg: SendEmail) -> Result<(), AppError> {
    ctx.app::<EmailService>().send(msg).await
}

// ── CronJob：触发器 = 时刻，In = tick ──
#[cron("0 0 2 * * *")]
async fn nightly_settle(ctx: &Ctx, _: CronTick) -> Result<(), AppError> {
    ctx.queue::<SettleTopic>().publish(SettleDay { day: ctx.now().yesterday() }).await
}
```

四个宏的**形状完全一致**：`async fn name(ctx: &Ctx, input: In) -> Result<Out, AppError>`。开发者学一个会全部。

### 4.3 App Core 复用（业务只写一次）

承接 `docs/arch.md`：**业务逻辑住在 App Core（`#[service]` 或纯 struct），所有 host 都是薄壳**。

```
  Route(api)  ─┐
  Route(ssr)  ─┤
  Job         ─┼──→ ctx.app::<OrderService>().create(...)   ← 业务唯一真相源
  Cron        ─┤        │
  RPC         ─┘        └─→ ctx.store::<Order>() / ctx.queue::<T>() / ctx.cache::<C>()
```

`create_order` 这个 API route 不写业务，只解请求 → 调 service → 编响应。同一个 `OrderService.create` 被 API、SSR、job 共享。

### 4.4 Ctx：请求级 vs 应用级（复用 RequestRuntime）

```rust
pub struct Ctx {
    app: AppRuntime,        // 应用级:service 图、executor 句柄(长生)
    req: RequestScope,      // 请求级:RequestRuntime(thread-local 隔离),含 auth/trace-id
}
impl Ctx {
    pub fn app<S: Service>(&self) -> S { S::deps(self) }          // 取 service
    pub fn store<E: SqlEntity>(&self) -> Store<E>;                 // 取 ORM 句柄
    pub fn cache<C: CacheResource>(&self) -> Cache<C::K, C::V>;    // 取缓存句柄
    pub fn queue<T: QueueResource>(&self) -> Producer<T::Msg>;     // 取队列生产者
    pub fn now(&self) -> Timestamp; pub fn auth(&self) -> &Auth;
}
```

**依赖注入 = typed handle**：`ctx.store::<Order>()` / `ctx.queue::<EmailTopic>()` 不只是取句柄，它们的**类型参数就是依赖声明**（§5 据此连边）。Web host 每请求 `with_request_runtime`；Worker/Cron host 每次执行包等价的 `with_job_runtime`，保证 thread-local 隔离 + panic 安全（复用 v1 的 RAII guard）。

### 4.5 多 Host 运行（复用 serve/serve_axum 装配点）

同一个二进制按**角色**启动不同 host（`rui run --role=web|worker|cron`），全部复用 v1 的装配点：

```rust
fn main() {
    let app = AppRuntime::boot();   // 注入 executor(连接信息来自部署期绑定,§5.4)
    match role() {
        Role::Web    => WebHost::new(app).serve_axum(),   // 复用 v1 serve_axum
        Role::Worker => WorkerHost::new(app).run(),       // 拉取/接收 topic → dispatch job
        Role::Cron   => CronHost::new(app).run(),         // 内置调度器 触发 → enqueue job
        Role::Rpc    => ServiceHost::new(app).serve(),    // service 的 RPC 暴露
    }
}
```

`WebHost` 直接是 v1 的 `serve_axum`（SSR/CSR/API route 都挂在 axum router 上，API route 走类似 `/graphql` 的同步引擎 seam）。`WorkerHost`/`CronHost`/`ServiceHost` 是新的薄 host，但共享 `AppRuntime` 和 `Ctx` 模型。

> **同步引擎的现实约束（审查 HIGH）**：v1 GraphQL 引擎 + postgres 驱动是同步的。v2 的 `async fn` 签名是**面子上的统一**；第一版 Worker/Cron host 用"线程池 + `spawn_blocking` 桥"执行（如 v1 axum host 已做），`async` 在 IO 边界 `block_on`。真正的 async 引擎重写推迟（§9）。统一的是**接口形状**，不是底层执行模型 —— 这点文档明确不藏。

### 4.6 Route 三态如何与 v1 router! 统一

- **SSR Route**：就是 v1 的 `#[page(ssr, "/x")]` + `router!`，`Strategy::Ssr`。`#[route(ssr,..)]` 是其语法糖别名。
- **CSR Route**：`Strategy::Csr`，v1 已有。
- **API Route**：新增 `Strategy::Api`，不渲染 HTML，handler 返回 `Json<T>`，挂在 axum router（或 std serve）上，复用 `/graphql` 的请求解析 seam。

三者共用一张 `router!`/`platform!` 路由表，`Strategy` 枚举多一个 `Api` 变体即可。**对 DAG**：每个 route 是一个节点，其 handler 里的 `ctx.app::<S>()` / `ctx.store::<E>()` 调用形成出边。

## 5. Pillar 3 — Compile → DAG → Deploy

这是 v2 的"奇迹发生处"，也是最需要把 Rust 可行性讲透的地方。

### 5.1 编译期 application model 提取机制

#### 5.1.1 头号难题：Rust 没有全局表

Rust proc-macro **只能看到它被附加的那一个 item**，看不到整个 program（memory 反复记录的约束）。v1 已经用两招绕过：`app!{}`/`router!{}` **显式装配**（用户列出所有页面）、`module_path!()` 做稳定 id。v2 必须把"提取整张应用拓扑"建立在同样诚实的基础上。

#### 5.1.2 选定主路线：inventory 分布式收集 + platform!{} 校验装配（双保险）

**机制 A（叶子事实收集）：`inventory`/`linkme` distributed slice。** 每个 `#[ent]`/`#[cache]`/`#[queue]`/`#[service]`/`#[route]`/`#[job]`/`#[cron]` 宏在 native 端 emit 一条 `inventory::submit!`，把"本 item 的局部事实"注册进一个全局 slice。链接期，所有分散的注册项被收集到一起 —— **这是 Rust 里事实上的"全局表"**，且跨 crate 有效（只要 crate 被链接进最终 binary）。

```rust
// 每个原语宏 emit 的节点描述(纯数据,可被 build 期读取)
inventory::submit! { Node::Service { name:"OrderService", deps:&[Dep::Store("Order"), Dep::Queue("emails")] } }
inventory::submit! { Node::Route { name:"create_order", method:"POST", path:"/orders", calls:&["OrderService"] } }
inventory::submit! { Node::Job { name:"send_email", topic:"emails", calls:&["EmailService"] } }
```

**机制 B（完整性校验 + 显式入口）：`platform!{}` 顶层装配宏。** 沿用 `router!` 哲学，要求用户在一处声明 platform 组成。它**不是**唯一真相源（那样太啰嗦），而是 **(1) 生成 `main`/`AppRuntime::boot`、(2) 对 inventory 收集到的节点做完整性断言**（声明了但没注册、注册了但没声明 → 编译期/启动期报错）。

```rust
rui::platform! {
    ents:   [User, Order, Role],
    caches: [UserCache],
    queues: [EmailTopic, SettleTopic],
    services: [OrderService, EmailService],
    routes: [create_order, order_page, orders_csr],
    jobs:   [send_email, settle_day],
    crons:  [nightly_settle],
    cloud:  Aws { region: "us-east-1" },   // 或 Gcp { project: .. }
}
```

> **为什么是双保险而非单靠 platform!{}**：纯靠 `platform!{}` 列举，宏依然读不到 `OrderService` 内部"用了哪些资源"（宏只看到 `OrderService` 这个 token）。所以**边**必须靠机制 A（节点自己 emit 自己的 deps）。`platform!{}` 负责"节点清单 + 完整性"，inventory 负责"边 + 细节"。两者交叉校验。

#### 5.1.3 边怎么被捕获：从 typed handle 到 Dep（诚实的约束与解法）

**审查 BLOCKER**：宏阶段**拿不到类型信息** —— `#[service] impl OrderService { fn deps(c:&Ctx)->Self { Self{ db: c.store::<Order>() } } }`，`#[service]` 宏看到的只是 token 流，它**无法知道** `c.store::<Order>()` 里的 `Order` 是个 Ent。怎么办？三层解法，按可靠性排序：

1. **显式 deps 标注（v2 第一版采用，最可靠）**：`#[service]` 宏扫描 `deps()` 函数体里的 `c.store::<T>()` / `c.queue::<T>()` / `c.cache::<T>()` / `c.app::<T>()` **调用模式**（这是纯语法匹配，宏能做），把泛型实参 `T` 的**路径名**抽出来当依赖。这不需要类型解析，只需要约定"依赖必须通过 `c.<kind>::<T>()` 取得"。

```rust
// 宏对 deps() 做语法扫描 → 抽出 [Store(Order), Queue(emails)]
fn deps(c: &Ctx) -> Self {
    Self { db: c.store::<Order>(), jobs: c.queue::<EmailTopic>() }
}
// emit: inventory::submit!{ Node::Service{ name:"OrderService", deps:&[Dep::Store("Order"), Dep::Queue("EmailTopic")] } }
```

2. **类型化字段 + 运行时自注册（兜底）**：`Store<E>`/`Queue<T>`/`Cache<C>` 的构造（`c.store::<E>()`）在运行时也向 RequestRuntime 记录一次"被访问的资源"。`rui build` 的 describe 阶段（见 5.3）实际**运行**一次 boot + 干跑各 handler 的 deps()，收集真实访问 —— 作为对语法扫描的交叉验证。

3. **`module_path!` + 名称解析在 describe 期做**：依赖名是字符串（"Order"），在 describe 期对照 inventory 里的 Ent 节点解析成真实节点引用。名称冲突由 `platform!{}` 的唯一性断言挡掉。

> **诚实的限制**：如果用户绕过 `c.store::<T>()` 约定（比如把 handle 存进一个 `Vec<Box<dyn Any>>` 再动态取），静态扫描会漏。v2 的立场：**约定优先**（idiomatic 写法必须用 typed `ctx` 取资源），漏检在 `rui build` 的图校验阶段通过"运行时 describe 干跑"兜底，并对"声明了 executor 却没有任何静态边"发 warning。

### 5.3 `rui build` / `rui deploy` 两阶段 CLI

```
rui build                    # 阶段一:编译 + 提取 model + 校验图
  ├─ cargo build --release   #   正常编译(inventory 注册项进 binary)
  ├─ 链接 describe harness    #   编译一个 --bin __rui_describe(platform!{} 生成)
  ├─ run __rui_describe       #   运行它:遍历 inventory slice + 干跑 deps() → dump .rui/model.json
  └─ validate(model)          #   图校验(见 5.4 红利):报环/孤儿/缺资源

rui deploy --env=prod        # 阶段二:model → 云
  ├─ read .rui/model.json
  ├─ provider = CloudProvider::from(model.cloud)   # Aws / Gcp
  ├─ plan = provider.plan(model)    # model 每个 Node → ResourceSpec;diff 现有状态
  ├─ 展示 plan(+N -M ~K)  并询问确认
  ├─ provider.apply(plan)           # provision DB/Cache/Queue/Secrets/计算
  ├─ build & push 容器镜像(多角色单镜像)
  ├─ 部署各 Host 角色(web/worker/cron) + 注入运行时绑定(5.4)
  └─ 健康检查 + 输出 endpoints
```

> **describe harness 的取舍**：相比 Encore 的独立 syn 源码解析器，rui 用"编译进 binary 的 inventory + 干跑"路线。**优点**：100% 复用 rustc 的类型检查与 cfg 处理，不重新实现 Rust 解析；跨 crate 天然工作。**缺点**：describe 需要能 boot（但用 mock executor，不连真实云）；dead-code 消除可能删掉没被引用的 inventory 项 —— 用 `linkme` 或对 `platform!{}` 列举的项强制 `#[used]` 引用规避。**退路**：若 inventory 在某 target 不可用，`platform!{}` 的显式列举本身已是一份"节点清单"，可降级为"用户列举节点 + 宏 emit 每节点的 deps 常量"，build 期直接读常量、不需运行 describe。

### 5.4 编译期校验红利 + CloudProvider

#### 编译期/build 期能静态报的错（"组件都是我们自己的"核心卖点）

- 用了未声明的资源：`ctx.store::<Ghost>()` 但 `Ghost` 不是 `#[ent]` → 编译错（trait bound 不满足）。
- Job 无消费者 / Topic 无消费者：`EmailTopic` 没有任何 `#[job(topic=EmailTopic)]` → build warning。
- Route 调了未注册 service：`create_order` 调 `OrderService` 但它没进 `platform!{}` → build error。
- 环依赖：service A↔B 互调 → build error（DAG 检测环）。
- 孤儿资源：声明了 `Cache` 但无人读写 → warning。
- Cron 表达式非法 / dedupe_key 模板引用了不存在的 payload 字段 → 编译错。

#### CloudProvider 抽象

```rust
// crates/rui-cloud/src/lib.rs(纯接口,被 rui-cli 用)
pub trait CloudProvider {
    fn plan(&self, model: &AppModel) -> ProvisionPlan;
    fn apply(&self, plan: &ProvisionPlan) -> Result<Deployment, CloudError>;
    fn destroy(&self, env: &str) -> Result<(), CloudError>;
    fn bind(&self, dep: &Deployment) -> RuntimeBindings;  // 产出连接信息
}
pub enum ResourceSpec {
    Compute { role: Role, cpu: f32, mem_mb: u32, min: u32, max: u32 },
    SqlDb   { engine: "postgres", size: .. },
    KvCache { ttl: .. },
    PubSub  { topic: String, ordering: bool },
    Bucket  { .. }, Scheduler { cron: String, target: String }, Secret { name: String },
}
```

#### AWS / GCP 资源映射表

| rui 逻辑资源 | AWS | GCP |
|---|---|---|
| Service / Web / Worker（计算） | ECS Fargate（或 App Runner） | Cloud Run |
| Cron 调度 | EventBridge Scheduler → 触发 | Cloud Scheduler → 触发 |
| Ent DB | RDS / Aurora Postgres | Cloud SQL Postgres |
| Cache | ElastiCache (Redis) | Memorystore (Redis) |
| Queue/Topic | SNS + SQS（pub/sub + 队列） | Pub/Sub |
| Job worker 消费 | SQS consumer（pull，ECS 常驻） | Pub/Sub **push** → Cloud Run |
| 静态资源 / CSR | S3 + CloudFront | GCS + Cloud CDN |
| Secrets | Secrets Manager | Secret Manager |
| 对象存储 | S3 | GCS |

> **模型差异（审查 HIGH）**：AWS SQS 是 **pull**、GCP Pub/Sub 典型是 **push**。统一的 `QueueExecutor` 抽象 pull 语义；GCP 适配器用一个内置的 push→pull 适配 endpoint（Cloud Run 收 push → 喂给 WorkerHost 的 dispatch）。Cloud Run 无常驻进程的限制 → Cron host 在 GCP 上用 Cloud Scheduler 触发，而非常驻调度器。这些 provider 差异**封死在 rui-cloud-{aws,gcp} 内**，不泄漏到用户代码。

#### provision 机制：自生成 Terraform（推荐）

**推荐生成 Terraform/OpenTofu**（而非直接 SDK 调用）：复用其成熟的 state 管理、diff/plan、幂等、回滚生态，rui 只负责"model → .tf"翻译 + 包装 `terraform apply`。**理由**：自己用 SDK 重写 state/diff/回滚是巨大且易错的工程；Terraform 是这块的事实标准。**退路**：对不喜欢 TF 的用户，`bind`-only 模式（你自己 provision，rui 只做运行时绑定）。

#### 运行时绑定（部署产物 → 注回 v1 注入点）

provision 完成后，连接信息（DB URL、cache endpoint、queue ARN、secrets ref）通过**环境变量 / secret 挂载**注入容器。`AppRuntime::boot()` 读取它们，调用 v1 的注入点：

```rust
fn boot() -> AppRuntime {
    let b = RuntimeBindings::from_env();          // 部署期写入的 env/secret
    set_store_executor(PgStore::connect(&b.database_url));     // v1 set_db_executor 同构
    set_cache_executor(RedisCache::connect(&b.cache_url));
    set_queue_executor(SqsQueue::connect(&b.queue_arn));       // 或 PubSubQueue(gcp)
    AppRuntime { .. }
}
```

**secrets 安全**：secrets 不进镜像、不进 model.json，只在运行时从 Secrets Manager 注入；IAM 最小权限由 provider 适配器按 DAG 边生成（service 只能访问它 DAG 上连到的资源）。

## 6. 端到端 worked example

下面用一个最小但完整的 "订单 + 欢迎邮件 + 每日结算" 应用，把 v2 的统一原语和一键部署串起来。它包含：2 个 Ent（`User`/`Order`）、1 个 Cache（`user_cache`）、1 个 Queue（`EmailTopic`）、2 个 Service（`OrderService`/`EmailService`）、Route 三态各一、1 个 AsyncJob、1 个 CronJob。

### 6.1 应用源码骨架

```rust
// ───────────────── domain/ : Ent 存储声明(同构) ─────────────────
// app/src/domain/user.rs
#[ent(table = "users", cache = "user_cache")]
pub struct User {
    #[ent(pk)] pub id: Id<User>,
    #[ent(index, unique)] pub email: String,
    pub name: String,
    #[edge(has_many = Order, via = "user_id")] pub orders: Edge<Order>,
}

// app/src/domain/order.rs
#[ent(table = "orders", mixins(Timestamps))]
pub struct Order {
    #[ent(pk)] pub id: Id<Order>,
    pub user_id: Id<User>,
    pub amount_cents: i64,
    pub status: OrderStatus,
}

// app/src/domain/cache.rs
#[cache(name = "user_cache", ttl = "5m")]
pub struct UserCache(Cache<Id<User>, User>);

// app/src/domain/queue.rs
#[queue(name = "emails", dedupe = true)]
pub struct EmailTopic(Topic<SendEmail>);
#[derive(Serialize, Deserialize)]
pub struct SendEmail { pub user_id: Id<User> }

// ───────────────── core/ : 业务只写一次 ─────────────────
// app/src/core/order_service.rs
#[service]
impl OrderService {
    fn deps(c: &Ctx) -> Self {
        Self { orders: c.store::<Order>(), users: c.store::<User>(), emails: c.queue::<EmailTopic>() }
    }
    pub async fn create(&self, input: CreateOrder) -> Result<Id<Order>, AppError> {
        let id = self.orders.insert(Order::new(input.user_id, input.amount_cents)).await?;
        self.emails.publish(SendEmail { user_id: input.user_id }).await?;   // → 边:Service→Queue
        Ok(id)
    }
    pub async fn get(&self, id: Id<Order>) -> Result<Order, AppError> { self.orders.find(id).await }
}

// app/src/core/email_service.rs
#[service]
impl EmailService {
    fn deps(c: &Ctx) -> Self { Self { users: c.store::<User>(), cache: c.cache::<UserCache>() } }
    pub async fn send(&self, m: SendEmail) -> Result<(), AppError> {
        let u = self.cache.get_or_load(m.user_id, || self.users.find(m.user_id)).await?;
        /* 发送邮件 */ Ok(())
    }
}

// ───────────────── api/ : 薄壳 host(Route 三态) ─────────────────
#[route(api, "POST /orders")]
async fn create_order(ctx: &Ctx, input: CreateOrder) -> Result<Json<Id<Order>>, AppError> {
    Ok(Json(ctx.app::<OrderService>().create(input).await?))
}
#[route(ssr, "/orders/:id")]
async fn order_page(ctx: &Ctx, p: PathParams) -> Result<Html, AppError> {
    let o = ctx.app::<OrderService>().get(p.id).await?;
    Ok(Html(view! { <main><h1>"Order " {o.id}</h1><p>{o.amount_cents}</p></main> }))
}
#[route(csr, "/orders/new")]              // 纯客户端壳
async fn new_order_page(_: &Ctx, _: ()) -> Result<Html, AppError> { Ok(Html(view!{ <NewOrderForm/> })) }

// ───────────────── jobs/ + cron/ : 薄壳 host ─────────────────
#[job(topic = EmailTopic, retries = 5, dedupe_key = "welcome:{user_id}")]
async fn send_email(ctx: &Ctx, msg: SendEmail) -> Result<(), AppError> {
    ctx.app::<EmailService>().send(msg).await    // → 边:Job→EmailService
}
#[cron("0 0 2 * * *")]
async fn nightly_report(ctx: &Ctx, _: CronTick) -> Result<(), AppError> {
    let n = ctx.store::<Order>().count_since(ctx.now().yesterday()).await?;
    println!("昨日订单 {n}"); Ok(())
}

// ───────────────── 一行装配 + 云目标 ─────────────────
// app/src/main.rs
rui::platform! {
    ents:   [User, Order],
    caches: [UserCache],
    queues: [EmailTopic],
    services: [OrderService, EmailService],
    routes: [create_order, order_page, new_order_page],
    jobs:   [send_email],
    crons:  [nightly_report],
    cloud:  Aws { region: "us-east-1" },
}
```

这就是**全部**。没有 Dockerfile、没有 Terraform、没有手连 RDS/SQS、没有手写 worker 主循环。

### 6.2 compile 出的 DAG

`rui build` 提取出：

```
  Route(api)  create_order ──Calls──▶ OrderService ──Writes──▶ EntDb(Order)
  Route(ssr)  order_page   ──Calls──▶ OrderService ──Reads───▶ EntDb(Order)
  Route(csr)  new_order_page (静态壳,无后端边)
                                      OrderService ──Publishes─▶ Queue(emails)
  Queue(emails) ──Consumes──▶ Job(send_email) ──Calls──▶ EmailService
                                                EmailService ──Reads──▶ EntDb(User)
                                                EmailService ──Reads──▶ Cache(user_cache)
  Cron(nightly_report) ──Triggers(02:00)──▶ self ──Reads──▶ EntDb(Order)

  推断出的基础设施: 1×Postgres(users,orders) · 1×Redis(user_cache) ·
                    1×SNS+SQS(emails) · 1×Web计算 · 1×Worker计算 · 1×Cron调度
```

## 7. crate / 模块结构

v2 在 v1 现有 crate（`rui` / `rui-macros` / `rui-macros-tests` / `rui-cli` / `examples/stocks`）基础上**增量演进**。核心原则：**`rui` 本体的依赖树永远只有 `rui-macros`**；所有 native/云/存储实现都是独立 optional crate，编译 wasm 客户端或零配置时一个都不拉。

```
crates/
  rui/                      # facade(不变):re-export 各层,prelude
    src/
      reactive.rs props.rs  #   L1 Kernel(同构,零依赖) —— 不动
      view.rs runtime.rs    #   L2 UI Runtime(同构) —— Route 三态在此
      gql/                  #   L3 Data Runtime(同构类型 + native executor)
        orm.rs              #     StoreExecutor(扩展 Query/Write:In/joins/batch)
        store_kv.rs  (新)   #     CacheExecutor 接口
        queue.rs     (新)   #     QueueExecutor 接口 + Topic/Producer 类型层
        exec.rs store.rs    #     + DataLoader(挂 RequestRuntime)
      server.rs server_axum.rs  # L4 Host(native) —— WebHost 复用
  rui-macros/               # 全部 proc-macro
    src/
      ent.rs                #   #[ent]/#[cache]/#[queue](emit SqlEntity + GraphQL + inventory node)
      handler.rs            #   #[route]/#[service]/#[job]/#[cron](emit Handler + inventory node)
      platform.rs           #   platform!{}(emit boot/main + describe bin + 完整性断言)
      view.rs page.rs ...   #   v1 既有(view!/page/router! 等)
  rui-platform/    (新)     # 计算 host:WebHost/WorkerHost/CronHost/ServiceHost + Ctx/AppRuntime
  rui-ent-pg/      (新)     # PgStore : StoreExecutor(把 examples 的 PgExecutor 提升为正式 crate)
  rui-ent-redis/   (新)     # RedisCache : CacheExecutor
  rui-ent-sqs/ rui-ent-pubsub/ (新)  # SqsQueue / PubSubQueue : QueueExecutor
  rui-model/       (新)     # AppModel/Node/Edge 纯数据(serde),build 与 cli 共享
  rui-cloud/       (新)     # CloudProvider trait + ResourceSpec/ProvisionPlan
  rui-cloud-aws/   (新)     # AWS 适配:model→Terraform,资源映射,运行时绑定
  rui-cloud-gcp/   (新)     # GCP 适配
  rui-cli/                  # rui build / rui deploy / rui run --role(扩展现有 rui-cli)
  rui-macros-tests/         # trybuild 契约(扩展:platform!{} / #[service] deps 扫描)
examples/
  stocks/                   # v1 示例(保持可编,验证向后兼容)
  shop/            (新)     # §6 的 worked example(展示全部 v2 原语)
```

**依赖方向**：`rui-platform`/`rui-ent-*` → `rui`（用其接口）；`rui-cloud-*` → `rui-model`（不依赖 `rui` 运行时）；`rui-cli` → `rui-model` + `rui-cloud-*`。`rui` **不**依赖任何上面这些。`layering.rs` 扩展断言守护。

## 8. v1 → v2 迁移路线图

每个阶段独立可交付、独立过 6 层验证（native/wasm 构建 · cargo test · verify.mjs · hydrate.mjs · SSR 多路由并发）+ 只读对抗 review。**非破坏优先**：v1 的 `#[derive(Ent)]`/`router!`/`serve_axum`/examples 全程保持可编。

| 阶段 | 内容 | 破坏性 | 验证要点 |
|---|---|---|---|
| **V2-0 安全网** | 把 §2 不变量加进 `layering.rs`（新接口不 NAME 驱动）；建 `rui-model` 空壳 | 无 | 现有测试全绿 |
| **V2-1 Cache** | `CacheExecutor` 接口 + `rui-ent-redis` + `#[cache]` 宏 + 内存回退 | 无（纯新增） | 内存/Redis 双后端 live |
| **V2-2 Queue + Job** | `QueueExecutor` + `#[queue]`/`#[job]` + `WorkerHost`（spawn_blocking 桥） | 无 | 内存队列 enqueue→consume 端到端 |
| **V2-3 统一 Handler** | 抽 `Handler` trait + `Ctx`；`#[route]`/`#[service]` 宏；`#[page]` 成 `#[route(ssr)]` 别名 | **低**（router! 兼容保留） | examples/stocks 用旧宏仍编；shop 用新宏 |
| **V2-4 Cron** | `#[cron]` + `CronHost`（内置调度器） | 无 | cron tick → enqueue job |
| **V2-5 Ent 关系** | edges/mixins/hooks + DataLoader + `Filter::In` | 无（v1 Ent 是子集） | N+1 消除、嵌套 GraphQL |
| **V2-6 platform!{} + DAG** | `platform!{}` 宏 + inventory 收集 + `rui build` + describe harness + 图校验 | 无 | shop 的 model.json 正确、图校验报错用例 |
| **V2-7 CloudProvider** | `rui-cloud` + `rui-cloud-aws`（先 AWS）+ `rui deploy` → Terraform | 无 | 真实 AWS 部署 shop（或 localstack） |
| **V2-8 GCP** | `rui-cloud-gcp` + push→pull 适配 | 无 | GCP 部署 shop |

**examples/stocks 的平滑演进**：先不动（V2-0..2 纯新增）；V2-3 后可选地把它的 `#[page]` 留作 `#[route(ssr)]` 别名验证；最终在 `examples/shop` 里展示完整 v2，stocks 作为"v1 兼容性回归"长期保留。

## 9. 已知限制 / 未决问题 / 第一版范围收敛

诚实记录四轮对抗审查中确认的 BLOCKER/HIGH，以及 v2 第一版**只承诺到什么程度**。

### 9.1 必须正面承认的限制

1. **同步引擎 vs async 接口（HIGH）**：四原语的 `async fn` 签名是"接口统一"，但 v1 引擎 + postgres 驱动是同步的。第一版 Worker/Cron host 用线程池 + `spawn_blocking` 桥执行，IO 边界 `block_on`。**真正的 async 引擎重写不在 v2 第一版**。这意味着高并发 IO 密集 worker 的效率受限于线程池大小 —— 可接受，且与 v1 axum host 现状一致。

2. **静态边提取依赖"约定"（BLOCKER 已降级）**：编译期提取依赖边要求资源必须通过 `ctx.store::<T>()`/`ctx.queue::<T>()`/`ctx.cache::<T>()`/`ctx.app::<T>()` 的 typed 调用获得（宏做语法扫描）。绕过约定（动态 `Box<dyn Any>` 存取）会漏检。兜底：`rui build` 的 describe 干跑 + "声明 executor 却无静态边"warning。**这是设计取舍，不是 bug**：idiomatic 写法必须用 typed ctx。

3. **describe harness 需可 boot（MED）**：`rui build` 要运行 `__rui_describe` 来 dump model，它需要能 `AppRuntime::boot()`（用 mock executor，不连真实云）。若应用 boot 有副作用，需用 cfg 隔离。退路：降级为"platform!{} 列举 + 每节点 deps 常量"，build 期直接读常量、不运行。

4. **inventory + dead-code 消除（MED）**：未被引用的 `inventory::submit!` 可能被 linker 删除。用 `linkme` 或 `platform!{}` 对列举项强制 `#[used]` 引用规避。需在 V2-6 重点测试（尤其 release + LTO + `panic=abort`，与 v1 profile 一致）。

5. **AWS/GCP 模型差异（HIGH）**：SQS（pull）vs Pub/Sub（push）、Cloud Run 无常驻进程。差异封死在 `rui-cloud-{aws,gcp}` 内，但意味着**两个 provider 的运行语义不完全等价**（如 GCP worker 是 push 驱动、扩缩容行为不同）。文档明示，不假装完全透明。

6. **DB migration 的演进（HIGH）**：Ent schema → DB migration 第一版只支持"加表/加列/加索引"等安全变更，自动生成 + `rui deploy` 前执行。**破坏性 migration（改类型/删列/数据回填）不自动化**，需人工 migration 脚本。这是数据安全红线。

### 9.2 第一版范围收敛（v2.0 只承诺这些）

**承诺**：
- Ent 覆盖 DB + Cache + Queue，带 edges/mixins/hooks + DataLoader，默认 GraphQL。
- Route(SSR/CSR/API)/Service/Job/Cron 四原语统一接口 + App Core 复用。
- `rui build` 提取 DAG + 图校验（环/孤儿/缺资源/无消费者）。
- `rui deploy` 一键部署到 **AWS**（GCP 在 V2-8），经生成 Terraform，含 DB/Cache/Queue/计算/Cron/Secrets provision + 运行时绑定 + 安全 migration。

**明确不承诺（v2.0）**：
- 不做多云同时部署、不做跨云迁移。
- 不做 async 引擎重写（线程池桥够用）。
- 不做 service mesh / 分布式 trace 自动注入（只做基础日志 + health）。
- 不做破坏性 DB migration 自动化。
- 不做可视化拓扑编辑器（DAG 只输出 `model.json` + 文本/Graphviz）。
- 不做零停机蓝绿/金丝雀（第一版滚动更新 + 健康检查即可）。

### 9.3 关键未决问题（需进一步设计）

- **跨 service RPC 的传输**：service 间调用第一版走"同进程直调"（单体部署）。真正的跨进程 RPC（独立扩缩容）需要选定传输（gRPC? rui 自有协议?）—— 推迟到 service 真正需要独立部署时。
- **本地开发体验**：`rui dev` 如何用内存/容器模拟 DB/Cache/Queue（localstack? testcontainers?）—— 影响 DX，需专门设计。
- **DAG 干跑的副作用隔离**：describe 期 boot 的最佳实践（mock executor 注入时机）。
- **状态漂移**：用户在云控制台手改了资源，`rui deploy` 的 diff 如何呈现与 reconcile（依赖 Terraform state，但需要 UX）。

---

> **结语**：v2 的本质是把 v1 已经验证的四个 seam（依赖倒置 / RequestRuntime 隔离 / 显式装配 registry / 可配置宿主）从"框架内部技巧"提升为"面向用户的统一原语 + 编译期部署系统"。最大的技术赌注是 §5 的"inventory 收集 + platform!{} 校验 + 约定式静态边提取"——它不追求"魔法般完整"，而是"诚实地静态可提取 + 干跑兜底 + 编译期校验红利"。这条路线完全建立在 Rust 的现实约束（无全局表、宏看不到类型）之上，且不破坏 v1 的任何一条不变量。
